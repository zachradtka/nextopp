# Row-Level Security for tenant isolation

Going public turns NextOpp from "one operator's data" into many mutually-distrusting **Accounts** ([CONTEXT.md](../../CONTEXT.md)) sharing one database. Isolation was enforced purely in application code — every query in [opportunities.ts](../../src/lib/actions/opportunities.ts) manually appends `eq(opportunities.userId, userId)` — and that discipline already failed silently once: `getStatusHistory` shipped with no ownership check while its sibling `getComments` had one, and because the file is `"use server"` every export is a directly-callable RPC endpoint. We decided to add Postgres Row-Level Security as a database-enforced backstop on all tenant tables, so a forgotten `WHERE` clause can no longer leak one Account's data to another. App code keeps its filters; RLS is the safety net underneath, not a replacement for them.

## Considered options

- **App-layer discipline only (status quo).** Rejected: it already failed (`getStatusHistory`), and with strangers exercising every endpoint, one missing clause is a cross-tenant breach of sensitive job-search data — an unacceptable failure mode to leave guarded only by code review.
- **Forced-scope repository wrapper + cross-tenant test, no RLS.** A helper that every tenant query must pass through, plus a test asserting Account B cannot read Account A. Rejected *as the sole mechanism* because a raw query still bypasses the wrapper — it is still app-layer trust. Kept as a *complement*: the cross-tenant test ships regardless.
- **`FORCE ROW LEVEL SECURITY`, keep connecting as the Neon owner role.** Fewer moving parts in the request path. Rejected because the app would remain over-privileged (the owner can run DDL — drop tables, disable policies), and admin scripts would then also hit RLS and need explicit bypass machinery. A restricted role is the cleaner least-privilege posture.
- **Neon Authorize (JWT → `auth.user_id()` in policies).** Removes the `SET LOCAL` plumbing. Rejected: it targets Neon's serverless/HTTP driver and a JWT-issuing auth provider, whereas we run `node-postgres` + Auth.js sessions and test on in-memory PGlite. Adopting it means driver lock-in and losing local/CI parity.
- **Subquery (`EXISTS`) RLS policies on the child tables.** Avoids a schema change. Rejected in favour of denormalising `user_id` onto the child tables with a composite foreign key: uniform, directly-indexable policies across every tenant table, and the FK makes a child row's owner *structurally* unable to diverge from its parent's — which is the entire reason we are moving off app-layer discipline.

## Decision

- **Restricted app role.** The running app connects as a dedicated non-owner role with no `BYPASSRLS`, no DDL, and only `SELECT/INSERT/UPDATE/DELETE` grants on the tenant tables. `DATABASE_URL` becomes this role at runtime. Migrations (the Vercel build's `npm run db:migrate`) and admin scripts (`import`, `seed`, the Turso→Neon migration) use a *separate* owner connection string — they legitimately need to bypass RLS and run DDL. This splits the single `DATABASE_URL` the project uses today; the build and CLI workflows must be updated to pass the owner URL.
- **RLS on tenant tables only.** `opportunities`, `status_history`, `opportunity_comments`. The Auth.js adapter tables (`users`, `accounts`, `verification_tokens`) are managed by `@auth/drizzle-adapter`, are not addressed by `app.user_id`, and are out of scope here; the app role gets only the grants the adapter requires.
- **Tenant context via a transaction-scoped GUC.** Every tenant query runs through a single `withTenant(userId, fn)` helper that opens a transaction and issues `SET LOCAL app.user_id = <userId>` before running the work. The tenant DB API is reachable *only* through this helper. A connection-level `SET` is forbidden: the `pg.Pool` reuses connections, so a session-level GUC would leak one tenant's context into the next tenant's query. `SET LOCAL` inside an explicit transaction is compatible with Neon's pooled (pgbouncer transaction-mode) endpoint, Docker Postgres, and PGlite.
- **One uniform, fail-closed policy.** Every tenant table uses `USING (user_id = current_setting('app.user_id', true))` with the matching `WITH CHECK`. `current_setting(..., true)` returns `NULL` when the GUC is unset, which matches no rows — so a query that forgets `withTenant` returns nothing rather than everything.
- **Denormalised `user_id` on child tables + composite FK.** `status_history` and `opportunity_comments` gain a `user_id` column (set from the session at insert), backfilled for existing rows. A composite FK `child(opportunity_id, user_id) → opportunities(id, user_id)` (requiring `UNIQUE(id, user_id)` on the parent) makes `child.user_id != parent.user_id` impossible to persist.
- **Fix the leak that motivated this.** `getStatusHistory` gets the same `requireUserId()` + ownership treatment as `getComments`; RLS then backstops it.

## Consequences

- Every tenant query now incurs a `BEGIN / SET LOCAL / … / COMMIT` round trip rather than a bare statement. Acceptable at this app's scale, but it is a uniform tax and the `withTenant` wrapper is non-negotiable plumbing — partial adoption reintroduces the exact gap RLS exists to close.
- Two connection identities now exist (restricted app role vs. owner). Environment management, the Vercel build step, and local dev must keep both straight; getting the migration/admin path wrong surfaces as DDL-permission or "rows mysteriously invisible" errors.
- The two identities also map to two Neon endpoints. The app role uses the **pooled** (pgbouncer, transaction-mode) endpoint — compatible with `SET LOCAL` inside a transaction, and necessary because every tenant query now holds a connection for a full transaction while Vercel Fluid Compute (reused instances, concurrent requests) multiplies open connections against Neon's limit; the `pg.Pool` `max` should stay modest. The owner role (migrations, admin scripts) uses the **direct** endpoint, because pgbouncer transaction-mode breaks the session-level features and advisory locks migrations depend on.
- The cross-tenant integration test (on PGlite) is part of this decision, not optional: it asserts that an Account sees zero of another Account's rows across every tenant table and operation, and that an unset GUC is fail-closed. It is the regression guard against both app-layer *and* policy mistakes.
- RLS protects against cross-tenant reads/writes through the app role. It does **not** substitute for the still-required app-layer `WHERE user_id` filters (they keep result sets correct and indexes used) — RLS is defence in depth beneath them, not instead of them.
